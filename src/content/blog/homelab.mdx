My homelab initially started as a way to host a few small services at home, but it ended up teaching me a lot about backend engineering.

Most backend projects are clean when they only run locally. The API starts, the database is reachable, the environment variables are in one place, and if something breaks it is usually a code problem. A homelab is not that clean.

A service can be running but still unreachable. DNS can point at the wrong place. A reverse proxy can route to the wrong port. A Docker volume can exist but have the wrong permissions. A tunnel can be healthy while the upstream service is dead. GPU transcoding can work on the host but fail inside a container.

My setup is built around Proxmox, LXC containers, Docker Compose, Pi-hole, Cloudflare Tunnel, Tailscale, Portainer, Infisical, and a media/reading stack with Jellyfin, Tdarr, Sonarr, Radarr, qBittorrent, Prowlarr, Kavita and Calibre-Web-Automated. The individual services are not really the point. The interesting part is how they fit together: networking, storage, DNS, secrets, permissions, background workers, service boundaries and debugging across layers. This post is a quick look at what I built, what broke, and why running real services at home made backend systems feel much less abstract.

## The Problem

When i started building backend projects most of them lived in clean environments: a local API, a test database, and maybe a staging server. The code was the main focus, and if something broke it was usually a code problem. This is a great way to learn the basics of backend development, but it doesn't prepare you for the complexities of running real services. In a production-like environment, there are many more moving parts: DNS, reverse proxies, tunnels, persistent storage, container networking, secrets management, permissions, background jobs, and more. These are the things that can break in ways that are not immediately obvious, and they are the things that you need to understand if you want to run real services.

## Networking

Networking was the part of self hosting that felt most like backend engineering. I split service access into three categories: docker, local, private, and public. Docker services are only accessible from other containers on the same network. Local services are only accessible from the LAN and are not exposed to the internet. Private services are things that i want to be able to reach from outside of the network but not exposed publicly, Tailscale works well for this as it gives me access to my private network without having to open ports on my router. Public services are things i want to be able to reach from anywhere, for this I use Cloudflare Tunnel's which allow me to expose services to the internet without having to open ports on my router. This setup allows me to have a clear separation of services and access levels, which is important for security and organization.

## DNS

DNS was one of the last things i set up, initially i was just using static IP addresses to reach my services, this worked as long as the IP addresses didn't change, but it was not a scalable solution. I set up Pi-hole as my DNS server, which allows me to use hostnames instead of IP addresses to reach my services. This makes it much easier to manage and remember the addresses of my services, and it also allows me to easily change the IP addresses of my services without having to update all of my clients.

## Background Workers

My media processing stack behaves a lot like a backend system, with background workers that process media files and update the database. I have a few different background workers that handle different tasks: transcoding media files, downloading metadata, and updating the database. These workers run in the background and are triggered by events, such as a new media file being added to the library. This setup allows me to offload processing tasks from the main application and keep the system responsive.

My faviroute example of this is my Tdarr setup. Tdarr scans files, cerates jobs, sends them to workers and writes the output back to storage

# ADD Component

The most interesting part of setting up Tdarr was setting up GPU passthrough for transcoding. Adding GPU transcoding to a containerized environment was a challenge, but it was worth it for the performance gains. I had to make sure that the GPU was accessible from within the container and that the necessary drivers were installed.

## Things tha broke

The things that taught me the most about linux and self hosting were the things that broke. Some of the problems I ran into were:

- Privleged containers: I had to learn about the security implications of running containers with elevated privileges and how to mitigate those risks.
- Permissions: I had to learn about file permissions and how to set them correctly for my services
- Networking: I had to learn about networking concepts and how to troubleshoot network issues in a containerized environment.
- GPU passthrough: I had to learn about how to set up GPU passthrough for my containers and how to troubleshoot issues that arose from that setup.
- DNS resolution: I had to learn about how DNS resolution works and how to troubleshoot issues that arose from misconfigured DNS settings.

## Closing Thoughts

The main thing self hosting taught me is that backend engineering is not just about writing code, it's about understanding the entire system and how all the pieces fit together. Running real services at home forced me to deal with the complexities of networking, storage, permissions, and more. It taught me how to troubleshoot issues that are not immediately obvious and how to think about the system as a whole. I highly recommend self-hosting as a way to learn backend engineering, as it provides a hands-on experience that is hard to replicate in a clean development environment.
